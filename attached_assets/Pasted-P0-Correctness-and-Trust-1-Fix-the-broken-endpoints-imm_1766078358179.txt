P0: Correctness and Trust
1) Fix the broken endpoints immediately

GET /api/shelter/applications/pending/count SQL syntax error near IN

Likely causes:

WHERE status IN ('pending') built incorrectly (string concatenation)

Passing array incorrectly into SQL builder

Missing parentheses or quoting

Fix pattern:

If using Drizzle/Prisma/etc., use the ORM’s inArray() / in helpers—not string SQL.

Add a unit test with multiple statuses + empty array behavior.

GET /api/shelter/calendar/[object Object]

Cause: a Date object (or params object) is being concatenated into the URL.

Fix:

Use query params: /api/shelter/calendar?start=YYYY-MM-DD&end=YYYY-MM-DD

Validate server-side and return 400 with a clear error if missing/invalid.

Why this is P0: a CRM is only “best” if staff trust the numbers and trust navigation.

P0: Tasks Must Reflect Reality (No “0 tasks completed” days)

Fix:

Introduce taskEvents (or an audit log) and auto-complete rules:

When vaccine record added → complete relevant vaccine task

When pipeline moved → complete stage transition tasks

When stray hold expires and auto-moves → complete hold monitoring task(s)

UI change:

“Suggested” vs “Required”

“Auto-completed by system” vs “Completed by staff”

Result: staff feel progress, and trust increases.

P1: Inbox Excellence (Your CRM moat)
4) Make the Inbox truly unified with one canonical object

You’re currently “merging” applications and conversations client-side. That will break at scale and complicate filtering.

Fix: Create a server-side inbox_items view (or endpoint)

One endpoint: GET /api/shelter/inbox

Each item includes:

type (application|conversation|system)

dogId, applicantId

lastActivityAt

unreadCount

priority, status, assigneeId

snippet

Why: this makes search, pagination, sorting, and unread tracking correct and fast.
